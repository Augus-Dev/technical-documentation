Introduction to Git

Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency.

Git is easy to learn and has a tiny footprint with lightning-fast performance. It outclasses SCM tools like Subversion, CVS, Perforce, and ClearCase with features such as cheap local branching, convenient staging areas, and multiple workflows.

To initialize a new local Git repository, use the command:

Command git init
  • Item 1: Advantages of Git over centralized systems.
  • Item 2: The distributed nature means every developer has a full copy of the project history.
  • Item 3: The "staging area" is a key feature.
Basic Commands

After initializing the repository, you need to add files to the staging area and then commit the changes.

To add a specific file, use:

Command git add [filename]

To add all modified files at once, use:

Command git add .

Committing is the act of permanently recording changes in the project history:

Command git commit -m "Descriptive message"

You can frequently view the current status of your repository using:

Command git status
Working with Branches

Branching allows you to deviate from the main line of development and work on new features or fixes without affecting the main code.

To create a new branch and switch to it immediately:

Command git checkout -b [branch-name]

To list all local branches:

Command git branch

When you finish work on a branch and want to integrate the changes into `main` (or `master`), you use `merge`:

Command git merge [branch-name]
  • Item 4: Merging can cause conflicts that need manual resolution.
GitHub and Remotes

GitHub is a code hosting platform that allows collaboration. You need to link your local repository to a remote repository on GitHub.

Add a new remote using a URL:

Command git remote add origin [repository_url]

Push your local changes to GitHub (main branch):

Command git push origin main

Pull changes from the remote to the local machine:

Command git pull origin main
Pull Requests and Collaboration

A Pull Request (PR) is the standard way to collaborate on GitHub. It is a request for the code from one branch to be reviewed and integrated into another branch.

Generally, it involves:

  • Item 5: Creating a feature branch.
  • Item 6: Committing and pushing to GitHub.
  • Item 7: Opening a PR via the GitHub web interface.

Code review is an essential part of the PR process, ensuring code quality.

Conflict Management and Rebase

Merge conflicts occur when two branches modify the same lines in the same file, and Git doesn't know which version to keep.

You need to open the conflicted file and manually resolve the sections marked with `<<<<<<<`, `=======`, and `>>>>>>>`.

After editing the file to the desired state, you add and commit:

Command git add [conflicted_filename]

Rebase is an alternative to merge for integrating changes. It rewrites the project history by moving or combining a sequence of commits to a new base commit.

Command git rebase main
  • Rebase keeps the project history cleaner and more linear.
Ignoring Files and Global Config

Often, there are automatically generated files (logs, caches, `node_modules`) that shouldn't be tracked by Git. For this, a `.gitignore` file is created.

You can specify file patterns to ignore, one per line.

Example # Ignore temporary files *.log # Ignore dependency folder node_modules/

Git's global configuration defines your username and email for all your local repositories.

Command git config --global user.name "Your Name"

You can check your current configuration with:

Ver Comando: git config --list git config --list
  • `.gitignore` must be committed to the repository.
  • The global configuration is stored in your home directory.
